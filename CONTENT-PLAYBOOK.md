# TechTidy Content Playbook

**Version:** 1.0  
**Last Updated:** July 2026

---

# Purpose

The TechTidy Content Playbook defines the standards for creating educational content across the TechTidy platform.

Its purpose is to ensure every lesson is:

- Easy to understand
- Technically accurate
- Visually consistent
- Practical
- Accessible
- Actionable

Every article should help readers build genuine digital confidence.

---

# Content Philosophy

> **Technology is already complicated. Learning it shouldn't be.**

Every lesson should reduce complexity without reducing accuracy.

Readers should finish each lesson feeling more confident than when they started.

---

# Audience

TechTidy is written for:

- Individuals
- Small businesses
- Nonprofits
- Educators
- Public sector organizations
- Professionals wanting practical technology skills

Assume readers are intelligent but may not have a technical background.

Never write down to the reader.

---

# Writing Principles

## Explain Before Defining

Introduce an idea before using technical terminology.

Example:

❌

Databases use CRUD operations.

✅

Every application needs a way to create, update, and delete information.
These actions are commonly known as CRUD operations.

---

## Use Plain Language

Prefer:

computer

instead of

workstation

Prefer:

website

instead of

web property

Prefer:

save

instead of

persist

---

## One Idea Per Paragraph

Large paragraphs create cognitive fatigue.

Aim for:

2–4 sentences.

---

## Show Before Telling

Whenever possible:

Example

↓

Diagram

↓

Explanation

instead of

Long explanation

↓

Example

---

## Be Practical

Every lesson should answer:

Why does this matter?

How will I encounter this?

How can I use it?

---

# TechTidy Lesson Structure

Every educational article follows the same format.

---

## Hero

Contains:

- Category
- Title
- Summary
- Reading Time
- Difficulty
- Last Updated

---

## Introduction

Answer:

Why should I care?

Length:

2–3 paragraphs.

---

## Learning Objectives

Readers should know exactly what they will learn.

Example

After this lesson you will understand:

- What databases are
- Static vs dynamic data
- Real-world examples
- Why it matters

---

## Main Sections

Each section should answer one question.

Recommended order:

Heading

Explanation

Example

Diagram (optional)

Callout (optional)

---

## Visual Support

Whenever possible include:

- Diagrams
- Tables
- Comparisons
- Process flows
- Screenshots

Avoid decorative imagery.

---

## Summary

Reinforce the main lesson.

Avoid introducing new concepts.

---

## Key Takeaways

Required on every article.

Three to five concise points.

Example:

✓ Databases store information.

✓ Static data rarely changes.

✓ Dynamic data changes frequently.

---

## Continue Learning

Link to:

- Previous lesson
- Next lesson
- Related articles

Learning should continue naturally.

---

# Recommended Article Length

Quick Guide

500–800 words

---

Standard Lesson

1,000–1,800 words

---

Deep Dive

2,000–3,000 words

Only when necessary.

Never add length without adding value.

---

# Reading Level

Target:

Grade 8–10 reading level.

Avoid unnecessary jargon.

Explain acronyms the first time they appear.

Example:

Multi-Factor Authentication (MFA)

Microsoft Entra ID

Application Programming Interface (API)

---

# Tone of Voice

The TechTidy voice is:

- Friendly
- Calm
- Professional
- Encouraging
- Practical

Avoid:

- Buzzwords
- Fear tactics
- Marketing language
- Clickbait
- Excessive hype

---

# Code Examples

When using code:

Explain:

What it does.

Why it matters.

Keep examples short.

Highlight only the important lines.

---

# Diagrams

Use diagrams when they improve understanding.

Good examples:

Architecture

Flowchart

Comparison

Timeline

Hierarchy

Avoid unnecessary decoration.

---

# Tables

Use comparison tables whenever comparing concepts.

Examples:

Cloud Storage vs Local Storage

OneDrive vs SharePoint

Password Manager vs Browser Passwords

Static vs Dynamic Data

Tables should remain concise.

---

# Screenshots

Screenshots should:

- Be cropped
- Highlight relevant areas
- Include captions
- Stay current

Update screenshots when interfaces change.

---

# Accessibility

Every article must:

- Use proper heading hierarchy
- Include alt text
- Avoid colour-only meaning
- Support keyboard navigation
- Meet WCAG AA colour contrast

Accessibility is a requirement.

---

# Internal Linking

Every lesson should include links to:

Related concepts

Prerequisite lessons

Next recommended lesson

This creates learning pathways.

---

# Metadata Standards

Every article should contain:

Category

Difficulty

Reading Time

Author

Last Updated

SEO Description

Keywords

---

# SEO Guidelines

Titles should clearly describe the topic.

Prefer:

Understanding APIs

instead of

Everything You Need to Know About APIs

Write naturally.

Never keyword stuff.

---

# Callouts

Use only approved callout types.

Definition

Tip

Example

Warning

Avoid creating new callout styles.

---

# Images

Every image must answer:

Does this help someone understand the topic?

If not, remove it.

---

# Lesson Checklist

Before publishing, verify:

- Clear title
- Strong introduction
- Learning objectives included
- Consistent headings
- Practical examples
- Helpful visuals
- Key Takeaways
- Continue Learning section
- Metadata complete
- Accessibility checked
- Responsive layout verified
- Proofread

---

# Publishing Workflow

1. Select topic

2. Research

3. Create outline

4. Write first draft

5. Add examples

6. Add visuals

7. Review for clarity

8. Accessibility check

9. SEO review

10. Publish

11. Add internal links

12. Share through newsletter and social media

# Content Lifecycle

Every lesson should be reviewed at least once every 12 months.

Review for:

Accuracy

Broken links

Outdated screenshots

Product changes

Improved examples

Add "Last Updated" when significant revisions are made.

---

# Success Criteria

A successful TechTidy lesson should leave readers able to answer:

- What is it?
- Why does it matter?
- Where will I use it?
- What should I remember?
- Where do I go next?

If readers can answer those questions confidently, the lesson has achieved its goal.

---

# Editorial Standards

When creating new content, always ask:

- Is this useful?
- Is it accurate?
- Is it approachable?
- Is it visually clear?
- Is it consistent with the TechTidy Design System?
- Would a beginner understand it?

If the answer to any question is "no," revise before publishing.

---

# The TechTidy Standard

Every article should leave readers feeling:

> **"I understand this now."**

That is the measure of success.

Technology will continue to evolve.

TechTidy's role is to make that change understandable, practical, and approachable for everyone.
